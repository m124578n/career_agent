# 104 Job Tracker — Side Project 規劃文件

> 版本：v0.1（初版）
> 最後更新：2026-06-20

---

## 1. 專案概述

### 動機
解決自己找工作時的痛點：重複性的職缺篩選、履歷比對、求職信撰寫等流程，透過 AI 自動化。附帶目標是順便探勘有沒有比現職更好的機會。

### 目標用戶
**第一位用戶就是自己。** 這代表設計上可以完全照自己的找工作習慣走，不需要一開始就追求通用化。等流程跑順、確認有價值，再考慮要不要開放給其他人。

### 成功定義（MVP 階段）
- 能上傳履歷、設定目標，拿到一份「這份履歷對這個職位的優勢／待補強」分析
- 能自動爬一批 104 職缺，逐筆給出契合度排序
- 對想投的職缺，能生成一封堪用的求職信
- 整個流程比手動找工作省時間 —— 這是唯一真正的成功指標

---

## 2. 核心流程（一條 Pipeline）

```
上傳履歷 + 設定目標職位/薪資
        ↓
履歷診斷（優勢 / 待補強）
        ↓
AI 下關鍵字 → 爬 104（分批，每批 20 筆）
        ↓
逐筆契合度分析 → 排序推薦
        ↓
User 決定投不投
   ├─ 投 → AI 生成求職信
   └─ 公司要求官網投遞 → 標記提醒（不自動處理）
        ↓
往下一批（21~40…）
        ↓
（後續）反向匹配：誰看過我 → 抓他們的其他缺
```

## 4. 技術選型

| 層 | 選擇 | 理由 |
|----|------|------|
| 爬蟲 | **Playwright** | 有經驗；104 有反爬，Playwright 比較穩 |
| Backend | **FastAPI** | 最熟 |
| LLM | **Claude API / OpenAI** | 職缺契合度分析、求職信生成 |
| 資料庫 | **MongoDB** | 職缺結構不固定，document DB 較彈性 |
| Frontend | **Streamlit / Gradio**（MVP）| 快速搭，OSENSE 做過 Gradio 上手快。之後若要精緻化再換 |
| 部署 | **Azure Container Apps** | 順便練到精藤的技術棧 |

## 5. 開發路線圖

### 第一週 — 爬蟲打底
把 104 爬蟲做起來，能穩定抓到職缺資料存進 MongoDB。先確認資料拿得到、結構長怎樣，這是整條 pipeline 的地基。

### 第二週 — LLM 分析
- 履歷上傳 + 解析（M1）
- 履歷診斷（M2）
- 把履歷餵進去，逐筆分析職缺契合度與缺口（M4 的分析部分）

### 第三週 — UI + 部署
- 簡單 UI 串起整條流程（Streamlit/Gradio）
- 求職信生成（M5）+ 外部投遞提醒（M6）
- 部署到 Azure Container Apps

> 節奏抓「先有個能跑的東西，再慢慢擴充」。每週結束都該有一個能 demo 的狀態。

## 6. 技術風險與取捨（先記著，免得做到一半撞牆）

1. **「誰看過我」需要登入 104**
   這跟公開職缺爬蟲是兩種難度——要處理登入、session、可能的驗證碼，自動化較脆。建議排到後面，或先用半自動（你登入、它讀畫面）。

2. **104 反爬機制**
   抓取頻率與被擋處理要留意。「第一批只抓 20 筆」這種節制設計反而是優點，天然降低被擋風險。

3. **官網投遞不碰**
   各家官網表單格式不一，自動化投資報酬率低。標記提醒、手動處理是務實取捨。

4. **履歷客製化（招募方偏好）的真實性**
   猜 HR 偏好缺乏資料支撐，容易變成「聽起來很厲害但沒根據」的功能。先做有明確邏輯的關鍵字優化，偏好猜測列為實驗性 P2。
